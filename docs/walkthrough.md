# Predictive Spatial Graph Engine (Execution Pipeline)

The Predictive Spatial Engine is an ultra-fast, deterministic text generator constructed by bridging a multi-tagged sensory Python pipeline directly into a geometric Rust Spatial Topology. 

This engine is rigorously designed to outperform traditional statistical Transformers (LLMs) in structurally complex boundaries (e.g., Corporate Triage, Edge Processing, Gaming NPCs) because it operates on mathematically verifiable spatial links rather than hallucination-prone probability matrices.

## 1. The Core Architectural Philosophy
By physically separating Semantic Inference from Topological Generation, this design structurally bypasses the historic algorithmic cliffs of conversational AI:

* **100% Traceable Hallucination Defense**: The graph generates language solely by walking explicit sequential anchors pre-ingested during training. If a sentence pathway does not explicitly exist mathematically, the engine physically cannot traverse it.
* **Immunity to Context Amnesia**: Transformers statistically forget subjects if paragraphs lengthen. This engine explicitly forces an orchestrator to maintain a rolling `Sessional Active Memory Array`, acting as an infinite mathematical compass driving exactly which constraints the generated graph traversal must securely hug.
* **Chronological Tie-Breaking**: Traditional models blend fact timelines. By natively storing explicit `dated` constraints into the base Edge struct, the Predictive Graph inherently phases out patched or outdated knowledge by treating `Time` exactly like an emotional metadata trait during walk generation!

## 2. The Step-by-Step Functional Workflow

1. **Topological Classification**: A user submits a paragraph. The Python endpoint securely chips the paragraph into an isolated Queue of intents using a Sentence Tokenizer. Each sentence natively mapped into distinct `Intent`, `Tone`, `Domain`, and `Entity` targets.
2. **Pre-Execution Sanitization**: As the intent Queue loads, the Rust Reasoning Module executes an algorithmic pre-scan. If the user contradicts themselves (*"Reset my router, actually skip the reset"*), the module logically scrubs the obsolete intents prior to execution.
3. **The OOV Snapping Sequence**: If the target input contains an extreme, uningested typo (`"recieved"`), the Rust engine doesn't wait for a slow ML response. It generates a near-instantaneous 5-dimensional `[f32; 5]` lexical vector based on structural string properties and mathematically "snaps" its placement onto the nearest identically shaped node (`"received"`) within nanoseconds.
4. **The Polymorphic Walk**: Generation dynamically shifts modes based purely on the `Intent`:
   * If generating an explanation, the orchestration queries the **Topological Link Density** surrounding the node to mathematically enforce the allowed sentence limit.
   * Based on the Orchestrator's Sessional Stack, path confidences multiply exponentially if their metadata mathematically matches the required `Topic` or `Tone`.
   * Finally, for questions, the architecture explicitly breaks traditional grammar sequence patterns, physically executing A* pathfinding coordinates to bridge unconnected 3D query hubs into functional Answer hubs.
5. **Multi-Signal Validation (The Hallucination Abort)**: If the prompt contains multiple primary entities (`"bank"` and `"ATM"`), the Engine natively executes an A* trace verifying physical connection. If `"ATM"` does not exist topologically connected to `"bank"`, the Walker systematically aborts the query, guaranteeing zero LLM-style contextual hallucination.
6. **The Stylistic RAG Pass**: The raw, mathematically true string (`"The bank closes at 5PM."`) verified by the Engine is finally transmitted to a disconnected, lightweight conversational miniLLM wrapper. This LLM is instructed uniquely to format the raw fact into a fluid, stylistic chatbot response, perfectly decoupling logic extraction from conversational styling.
